# Contributing to macroverse-r

Thanks for taking an interest in macroverse-r. These packages are a small collection of R tools for international economic and social data analysis, and contributions of any size are welcome -- from typo fixes to new features.

This file applies across every macroverse-r repository unless the repo overrides it with its own `CONTRIBUTING.md`. It covers the macroverse-r conventions; for the underlying R-package development workflow it builds on, see the tidyverse development contributing guide at https://rstd.io/tidy-contrib.

## Reporting issues

Before opening an issue, search existing ones. If the problem is new, include:

- a minimal reprex (see https://www.tidyverse.org/help/#reprex),
- the package and version, the R version, and `sessionInfo()`,
- what you expected and what happened.

Bugs specific to one package belong in that package's repo. Cross-cutting concerns can be opened against `macroverse-r/.github`.

## Proposing changes

- Trivial fixes (typos, one-line renames) can go straight to a pull request.
- For anything larger, open an issue first describing the problem or motivation. This avoids wasted work and creates a searchable record of why a change was made. Link it from your PR with `Closes #N`.
- Keep each PR focused on a single theme; split unrelated work into separate PRs.

## Pull request process

- Fork the package repo (external contributors) or use a feature branch (maintainers). Never commit directly to `main`.
- Edit the `.R` source files, not the generated `man/*.Rd` or `NAMESPACE` -- those are regenerated by roxygen2.
- Write tests in `tests/testthat/` for the behavior you changed.
- Run `devtools::check()` -- it must be clean (0 errors, 0 warnings).
- Add a bullet to `NEWS.md` for user-facing changes: `* Description. (@username, #PR)`.
- Follow the tidyverse style guide (https://style.tidyverse.org) and run `styler::style_pkg()`.
- Wait until CI is green before requesting review.
- Fill in the PR template (it is set at the org level in the `macroverse-r/.github` repo and applies to every macroverse-r repo automatically).

## Review and merge

Reviews happen as maintainer time permits. If there is no response after about a month, feel free to ping -- it usually means the PR slipped past notice, not that it is being rejected.

Merge strategy is squash-and-merge: each PR lands on `main` as a single commit. History stays linear and one commit corresponds to one reviewed change. This matches the convention in the wider R ecosystem (tidyverse, r-lib, data.table).

## Code of Conduct

By participating in this project you agree to abide by the Contributor Covenant (https://www.contributor-covenant.org/version/2/1/code_of_conduct/). Be kind; assume good faith.
